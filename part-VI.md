# Часть VI. Аварийные сценарии и восстановление *(90 стр., 12 кейсов)*

> **Цель:** освоить полный цикл аварийного реагирования — от классификации отказов через моделирование 12 сценариев до post-mortem анализа.
> **После этой части вы сможете:** классифицировать отказы по доменам и моделям, отработать любой аварийный сценарий, спланировать Disaster Recovery, написать post-mortem.

## Глава 18. Типология отказов *(12 стр.)*

### 18.1. Домен отказа *(3 стр.)*
- DOT-схема: вложенные домены — диск → OSD → хост → стойка → DC
- Что отказывает на каждом уровне
- Как CRUSH обеспечивает изоляцию доменов

### 18.2. Модели отказов *(3 стр.)*
- Fail-stop: компонент явно остановился (OSD упал, диск отключился)
- Fail-slow: компонент работает, но медленно (latent disk, перегруженная сеть)
- Byzantine: компонент ведёт себя некорректно (bit flip, злонамеренный)
- Partition: сетевое разделение (split-brain, изоляция стойки)

### 18.3. RTO и RPO *(3 стр.)*
- Recovery Time Objective: сколько времени на восстановление
- Recovery Point Objective: сколько данных допустимо потерять
- Расчёт для Ceph: RTO (минуты–часы), RPO (0 при репликации ×3)

### 18.4. Стратегии *(3 стр.)*
- Repair: починить сломанный компонент
- Rebuild: пересоздать заново
- Failover: переключиться на резервный компонент
- Disaster Recovery: полное восстановление кластера

## Глава 19. Моделирование отказов: 12 кейсов *(42 стр.)*

> Каждый кейс: модель отказа → симптомы → хронометраж → восстановление → проверка.

### 19.1. Кейс 1: отказ одного OSD *(3 стр.)*
- Модель: `systemctl stop ceph-<fsid>@osd.X`
- Хронометраж: через сколько MON замечает (`mon_osd_down_out_interval`), degraded, backfill, active+clean
- DOT-схема: цепочка событий на временной шкале

### 19.2. Кейс 2: отказ узла с 4 OSD *(3 стр.)*
- Модель: `shutdown -h now` на OSD-узле
- Массовый recovery: `osd_max_backfills`, `osd_recovery_max_active`
- Влияние на клиентский трафик: замеры latency до/во время/после

### 19.3. Кейс 3: потеря MON (1 из 3) *(3 стр.)*
- Модель: `systemctl stop ceph-mon@<host>`
- Кворум жив (2/3): `HEALTH_WARN`, MON down
- Восстановление: запуск MON, синхронизация

### 19.4. Кейс 4: потеря MON (2 из 3) *(4 стр.)*
- Модель: остановка 2 MON одновременно
- Нет кворума: кластер не отвечает на запросы
- Ручное восстановление: извлечение monmap → формирование нового кворума → `ceph-mon --mkfs`

### 19.5. Кейс 5: split-brain сети *(3 стр.)*
- Модель: iptables-блокировка между половинами кластера
- Две половинки: каждая думает, что она — большинство
- Разрешение: снятие блокировки → перевыборы → синхронизация

### 19.6. Кейс 6: отказ стойки *(3 стр.)*
- Модель: отключение коммутатора стойки
- CRUSH failover: данные автоматически читаются с других стоек
- Перебалансировка: `ceph osd out` для OSD в упавшей стойке

### 19.7. Кейс 7: случайное удаление пула *(3 стр.)*
- Модель: `ceph osd pool rm <pool> <pool> --yes-i-really-really-mean-it`
- Защита: `pool_delete_requires_confirm_pools=2`
- Восстановление: снапшоты пула, RBD trash, бэкап monmap

### 19.8. Кейс 8: повреждение метаданных PG *(4 стр.)*
- Модель: ручная порча объекта на OSD (`dd` в raw-раздел)
- `ceph pg repair` — когда помогает
- Когда не помогает: `ceph-objectstore-tool export` → ручное восстановление

### 19.9. Кейс 9: повреждение monmap *(3 стр.)*
- Модель: удаление `/var/lib/ceph/mon/ceph-<host>/store.db/`
- Восстановление: `ceph-monstore-tool` → извлечение из другого MON
- DOT-схема: как MON синхронизируют monmap между собой

### 19.10. Кейс 10: min_size нарушен *(4 стр.)*
- Модель: pool size=3, min_size=2; остановить 2 OSD с общими PG
- Симптомы: PG не могут обслуживать запись, клиенты в блокировке
- Стратегия: `osd_failsafe_ratio`, `noout` при временных отказах

### 19.11. Кейс 11: восстановление из снапшотов *(4 стр.)*
- RBD снапшот: `rbd snap create`, `rbd snap rollback`
- CephFS снапшот: `mkdir .snap/<name>`, восстановление файла через `cp`
- Пошаговый сценарий: удалили файл → нашли снапшот → восстановили

### 19.12. Кейс 12: полный DR *(4 стр.)*
- Сценарий: потерян весь кластер (сгорел DC)
- Что сохранили: бэкап monmap, OSD-образы, ключи
- Новый кластер: `cephadm bootstrap` → импорт monmap → восстановление OSD
- Миграция клиентов: переключение RBD Mirror, CephFS standby

### 19.13. Практикум *(1 стр.)*
- Мини-DR: потеряли MON + OSD, восстанавливаем с нуля
- DOT-схема: план восстановления

## Глава 20. Бэкап и Disaster Recovery *(20 стр.)*

### 20.1. Что бэкапить *(3 стр.)*
- monmap: `ceph mon getmap -o monmap.bin`
- OSD map: `ceph osd getmap -o osdmap.bin`
- CRUSH map: `ceph osd getcrushmap -o crushmap.bin`
- Ключи: `ceph auth export`
- План бэкапа: автоматизация через cron + скрипт

### 20.2. RBD Mirror *(5 стр.)*
- Асинхронная репликация RBD-образов между кластерами
- DOT-схема: первичный кластер → вторичный кластер
- `rbd mirror pool enable`, `rbd mirror image enable`
- Failover: `rbd mirror image demote/promote`
- Мониторинг: `rbd mirror pool status`

### 20.3. CephFS Snapshots *(3 стр.)*
- `snap-schedule` модуль MGR: автоматические снапшоты
- Политика: каждый час (keep 24), каждый день (keep 7), каждую неделю (keep 4)
- Восстановление: `cp /mnt/cephfs/.snap/hourly.2026-07-07_14:00/file.txt .`

### 20.4. RGW Multi-site *(5 стр.)*
- Зоны, zone groups, realm (DOT-схема: мульти-DC топология)
- Sync policy: двунаправленная, однонаправленная
- Конфликты: как разрешаются (last-writer-wins)
- `radosgw-admin` для управления multi-site

### 20.5. Практикум: RBD Mirror *(4 стр.)*
- Два кластера: развернуть, настроить RBD Mirror
- Создать образ, записать данные, проверить репликацию
- Failover: demote → promote → переключение клиента

## Глава 21. Протоколирование и post-mortem *(16 стр.)*

### 21.1. Журнал аварии *(3 стр.)*
- Чек-лист на 20 пунктов: время обнаружения, симптомы, кто заметил, действия, результат
- Пример заполненного журнала по кейсу 19.1

### 21.2. Восстановление хронологии *(3 стр.)*
- `ceph log last [N]` — кластерный журнал
- `journalctl --since "2026-07-07 14:00" -u ceph-*`
- Аудит: кто и когда выполнял команды (`ceph tell` логируется)

### 21.3. Post-mortem анализ *(4 стр.)*
- DOT-схема: блок-схема процесса (5 шагов)
  1. Сбор фактов (timeline)
  2. Определение первопричины (5 whys)
  3. Оценка влияния (RTO/RPO, пользователи)
  4. План предотвращения
  5. Action items + владельцы + сроки

### 21.4. Шаблон post-mortem *(3 стр.)*
- Структура: Summary, Timeline, Root Cause, Impact, Prevention, Actions
- Пример заполненного шаблона по кейсу 19.4 (потеря MON majority)

### 21.5. Практикум: post-mortem *(3 стр.)*
- По кейсу 19.4 (потеря MON majority): написать post-mortem по шаблону
