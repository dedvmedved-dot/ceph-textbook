# Ceph: Учебное пособие для молодых специалистов

> **Целевая аудитория:** администраторы и разработчики с базовыми знаниями информатики (школьный уровень).
> **Подход:** от простого к сложному, от общего к частному. Каждый термин расшифровывается, каждая команда объясняется построчно.
> **Особые условия:** рассматривается развёртывание в замкнутом контуре (без доступа в Интернет).

## ОГЛАВЛЕНИЕ

### Часть I. Основы распределённого хранения данных

#### Глава 1. Зачем нужно распределённое хранение?
- 1.1. Как компьютер хранит данные: диски, файлы, блоки
  - *Практикум:* посмотреть свои диски через `lsblk`, понять разделы и файловые системы
- 1.2. Проблема: что делать, когда данных больше, чем диск?
- 1.3. Проблема: что делать, если диск сломался?
- 1.4. Идея распределённого хранения: много дисков на разных серверах
- 1.5. Ключевые понятия: репликация, шардирование, отказоустойчивость
  - *Схема:* один диск vs массив дисков vs распределённое хранилище
- 1.6. Термины: узел (node), кластер (cluster), Petabyte, IOPS, latency

#### Глава 2. Что такое Ceph и зачем он нужен?
- 2.1. Краткая история: Sage Weil, DreamHost, UCSC (2004–2006)
- 2.2. Философия Ceph: self-healing, self-managing, software-defined
- 2.3. Где применяется Ceph: CERN (100+ PB), CloudStack, OpenStack, Kubernetes
  - *Схема:* место Ceph в экосистеме хранения данных
- 2.4. Ceph vs другие системы: RAID, NFS, GlusterFS, MinIO — сравнение
  - *Таблица:* сравнительные характеристики (масштабируемость, отказоустойчивость, сложность)
- 2.5. Термины: SDS (Software-Defined Storage), CRUSH, RADOS, PG, OSD, MON

### Часть II. Архитектура Ceph: как это работает внутри

#### Глава 3. Общая архитектура Ceph
- 3.1. Четыре уровня Ceph: RADOS → librados → RBD/CephFS/RGW → приложения
  - *Схема:* слоёная диаграмма архитектуры
- 3.2. RADOS (Reliable Autonomic Distributed Object Store) — сердце Ceph
  - Что такое «объект» в Ceph? Чем объект отличается от файла и блока?
  - *Практикум:* создать объект и посмотреть его атрибуты
- 3.3. CRUSH (Controlled Replication Under Scalable Hashing) — алгоритм размещения
  - Как Ceph решает, на какой диск положить данные, без центральной таблицы?
  - *Схема:* CRUSH map → OSD
  - *Практикум:* посмотреть CRUSH map своего кластера, понять weight и placement
- 3.4. PG (Placement Group) — логическая группа объектов
  - Зачем нужны PG? Как они связаны с OSD?
  - Формула расчёта количества PG: `pg_num = (OSD_count × 100) / replica_count`
  - *Практикум:* рассчитать PG для своего кластера
- 3.5. Термины: OSD, MON, MDS, MGR, RGW, pool, replica, erasure coding

#### Глава 4. Компоненты Ceph — подробно
- 4.1. MON (Monitor) — мозг кластера
  - Что хранит MON? Cluster map, OSD map, PG map, MDS map, CRUSH map
  - Алгоритм Paxos: как MON-ы договариваются между собой?
  - *Схема:* Paxos — кворум, лидер, голосование
  - Почему MON должно быть нечётное количество? (3, 5, 7…)
  - *Практикум:* `ceph mon stat`, `ceph mon dump`
- 4.2. OSD (Object Storage Daemon) — рабочие руки кластера
  - Жизненный цикл объекта на OSD: запись → репликация → scrubbing → удаление
  - *Схема:* путь записи от клиента до диска
  - Физическое хранение: BlueStore (с 2017), FileStore (устаревший)
  - *Практикум:* заглянуть внутрь OSD: `ceph osd tree`, `ceph osd df`
- 4.3. MGR (Manager) — панель управления
  - Балансировка, PG autoscaler, dashboard, prometheus-метрики
  - *Практикум:* открыть Dashboard, посмотреть метрики
- 4.4. MDS (Metadata Server) — файловая система
  - Как CephFS хранит файлы: метаданные отдельно от данных
  - *Схема:* файл → inode → объекты данных
- 4.5. RGW (RADOS Gateway) — S3/Swift-совместимое объектное хранилище
  - *Практикум:* загрузить файл через S3 API (boto3)

#### Глава 5. Путь данных: от клиента до диска
- 5.1. Клиентская библиотека librados: как клиент общается с кластером
- 5.2. Шаг 1: клиент запрашивает cluster map у MON
- 5.3. Шаг 2: CRUSH вычисляет, на какой OSD писать
- 5.4. Шаг 3: клиент пишет напрямую на primary OSD (без центрального сервера!)
- 5.5. Шаг 4: primary OSD реплицирует на secondary OSD
- 5.6. Шаг 5: подтверждение клиенту
- 5.7. *Схема:* полный путь одного запроса с таймингами
- 5.8. *Практикум:* включить отладку на клиенте, посмотреть логи

### Часть III. Развёртывание Ceph

#### Глава 6. Планирование кластера
- 6.1. Как посчитать, сколько нужно дисков и серверов?
  - Формулы: сырая ёмкость → полезная ёмкость (с учётом репликации)
  - *Практикум:* рассчитать кластер на 100 TB полезного пространства
- 6.2. Сетевые требования: 10GbE/25GbE/100GbE, public vs cluster network
- 6.3. Выбор оборудования: HDD vs SSD vs NVMe, контроллеры, память
- 6.4. Требования к CPU и RAM для каждого типа узла
- 6.5. *Схема:* типовая стойка с Ceph-кластером

#### Глава 7. Ручное развёртывание (для понимания)
- 7.1. Установка Ceph вручную: шаг за шагом
  - `mkcephfs` (исторический метод) — просто чтобы понять компоненты
- 7.2. Запуск первого MON вручную
  - *Практикум:* создать MON вручную, посмотреть его файлы
- 7.3. Запуск первого OSD вручную
  - `ceph-osd -i 0 --mkfs`, `ceph-osd -i 0`
  - *Практикум:* создать OSD вручную на файле (loopback), понять структуру каталога
- 7.4. Ручное создание пула и запись объекта
  - `rados put`, `rados get`
- 7.5. Зачем это знать, если есть автоматизация? Понимание внутренностей для отладки.

#### Глава 8. Развёртывание через cephadm (промышленный стандарт)
- 8.1. Что такое cephadm? Docker/Podman-контейнеры, оркестрация
- 8.2. Подготовка узлов: установка Docker, настройка сети, пользователи
  - *Полный скрипт:* подготовка Ubuntu 24.04 для Ceph (построчный разбор)
- 8.3. `cephadm bootstrap` — что происходит внутри?
  - Разбор вывода команды: каждый этап
  - *Схема:* что создаётся при bootstrap (ключи, MON, MGR, конфиги)
- 8.4. Добавление узлов: `ceph orch host add`
  - Как cephadm подключается по SSH? Как распространяются ключи?
- 8.5. Размещение сервисов: `ceph orch apply mon/osd/mds`
- 8.6. *Практикум:* полный скрипт развёртывания с комментариями к каждой строке
- 8.7. Типичные ошибки при развёртывании и их решение
  - Docker socket activation, PG limits, mismatched releases

#### Глава 9. Развёртывание в замкнутом контуре (Air-Gapped)
