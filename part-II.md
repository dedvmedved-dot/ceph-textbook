# Часть II. Архитектура Ceph *(70 стр.)*

> **Цель:** разобрать внутреннее устройство Ceph до уровня «могу нарисовать на доске и объяснить каждую стрелку».
> **После этой части вы сможете:** рассказать, как RADOS хранит объекты, как CRUSH вычисляет размещение без центральной таблицы, и проследить путь клиентского запроса от приложения до диска.

## Глава 3. RADOS — фундамент *(24 стр.)*

### 3.1. Reliable Autonomic Distributed Object Store: разбор каждого слова *(3 стр.)*
- **Reliable** — надёжный: репликация, самовосстановление
- **Autonomic** — автономный: без ручного вмешательства
- **Distributed** — распределённый: нет единой точки отказа
- **Object Store** — объектное хранилище: не файлы, не блоки — объекты
- Что такое «объект» в Ceph: OID, данные, метаданные, xattrs

### 3.2. MON (Monitor): мозг кластера *(5 стр.)*
- Что хранит MON: cluster map (monmap, osdmap, pgmap, mdsmap, crushmap)
- Кворум: почему нечётное количество (3, 5, 7…), формула большинства
- Paxos: как MON-ы договариваются (алгоритм консенсуса — упрощённо)
- DOT-схема: кворум 3/5 — кто лидер, кто ведомый
- `ceph mon stat`, `ceph mon dump` — читаем вывод

### 3.3. OSD (Object Storage Daemon): рабочие руки *(5 стр.)*
- Функция OSD: хранение объектов, репликация, восстановление, scrubbing
- BlueStore: что внутри? BlockDevice → BlueFS → RocksDB (DOT-схема слоёв)
- WAL (Write-Ahead Log) и DB: зачем, где размещать
- Аллокатор BlueStore: как выбирается свободное место на диске
- `ceph osd tree`, `ceph osd df`, `ceph osd metadata`

### 3.4. MDS (Metadata Server): файловая система *(3 стр.)*
- Роль MDS: хранит только метаданные CephFS (не данные!)
- Журнал MDS: recovery после сбоя
- Subtree pinning: привязка каталогов к конкретному MDS
- Standby MDS: горячий резерв

### 3.5. MGR (Manager): панель управления *(3 стр.)*
- Модульная архитектура: `ceph mgr module ls`
- Ключевые модули: prometheus, dashboard, balancer, pg_autoscaler, zabbix
- Как MGR собирает метрики со всех OSD

### 3.6. Взаимодействие компонентов *(3 стр.)*
- DOT-схема: потоки управления (MON↔OSD↔MDS↔MGR) и потоки данных (клиент↔OSD)
- Кто с кем говорит, по какому протоколу, на каких портах

### 3.7. Практикум: «собери RADOS» *(2 стр.)*
- На работающем кластере: `ceph status`, `ceph osd tree`, `ceph mon dump`
- Сопоставляем вывод команд с теоретической схемой

## Глава 4. CRUSH — алгоритм размещения *(23 стр.)*

### 4.1. Controlled Replication Under Scalable Hashing *(2 стр.)*
- Перевод: контролируемая репликация через масштабируемое хеширование
- Ключевое свойство: псевдослучайное, но детерминированное размещение
- Зачем: клиент вычисляет OSD сам, без запроса к центральному серверу

### 4.2. Проблема 10⁹ объектов *(3 стр.)*
- Почему нельзя хранить таблицу «объект → диск» для миллиарда объектов
- Почему DHT (distributed hash table) не подходит: нет контроля над доменами отказа
- Решение CRUSH: хеш-функция + иерархическая карта = O(1) поиск

### 4.3. CRUSH map: иерархия, типы, веса *(5 стр.)*
- Декомпиляция: `crushtool -d` — читаем построчно
- Иерархия: `root` → `datacenter` → `rack` → `host` → `osd`
- Типы устройств (buckets): uniform, list, tree, straw, straw2
- Веса (weight): что они значат, как влияют на распределение данных
- `ceph osd crush tree` — читаем и понимаем

### 4.4. Правила CRUSH *(5 стр.)*
- Синтаксис правил: `rule` → `take` → `choose`/`chooseleaf` → `emit`
- 5 примеров правил:
  1. Репликация по хостам (3 копии, разные хосты)
  2. Репликация по стойкам
  3. SSD-only пул
  4. Erasure coding (k=3, m=2)
  5. Разделение по датацентрам
- Как правило связывается с пулом: `ceph osd pool get <pool> crush_rule`

### 4.5. Placement Groups *(4 стр.)*
- Что такое PG: логическая группа объектов (не физическая!)
- Зачем PG: уменьшить количество связей «объект → OSD»
- Формула: `pg_num = (OSD_count × 100) / replica_count`
- PG autoscaler: как работает, когда включать
- Состояния PG: active, clean, degraded, undersized, peered

### 4.6. Практикум: работа с CRUSH map *(4 стр.)*
- `ceph osd getcrushmap -o crush.bin`
- `crushtool -d crush.bin -o crush.txt`
- Редактируем: добавляем стойку `rack`, переносим OSD
- `crushtool -c crush.txt -o crush-new.bin`
- `ceph osd setcrushmap -i crush-new.bin`
- Проверяем: `ceph osd crush tree`

## Глава 5. Путь данных: от клиента до диска *(23 стр.)*

### 5.1. librados: клиент без шлюза *(3 стр.)*
- Клиентская библиотека: `librados` (C++), биндинги для Python, Java, Go
- Клиент не ходит через MON за каждым объектом — получает карту и кеширует
- Обновление карты: как клиент узнаёт об изменениях в кластере

### 5.2. Навигация без карты *(4 стр.)*
- DOT-схема: цепочка поиска OSD
  1. Клиент → MON: «дай последнюю карту кластера»
  2. MON → Клиент: monmap + osdmap + crushmap
  3. Клиент: `CRUSH(pool_id, object_hash) → [osd.3, osd.7, osd.12]`
  4. Клиент → osd.3 (primary): запрос напрямую

### 5.3. Запись: по шагам *(5 стр.)*
- DOT-временна́я диаграмма: клиент → primary → secondary → ack
- Шаг 1: клиент отправляет запись на primary OSD
- Шаг 2: primary пишет в свой журнал (WAL)
- Шаг 3: primary отправляет репликацию на secondary OSD
- Шаг 4: secondary пишут в свои журналы и подтверждают
- Шаг 5: primary подтверждает клиенту
- Когда данные попадают на диск, а когда — в журнал?

### 5.4. Чтение *(3 стр.)*
- Primary read: чтение с primary OSD (по умолчанию)
- Local read: чтение с ближайшего OSD (CRUSH locality)
- Балансировка чтения: `ceph osd set-require-min-compat-client`
- Degraded-режим: что происходит, когда одна из реплик недоступна

### 5.5. Scrubbing: самопроверка Ceph *(4 стр.)*
- Light scrub: проверка контрольных сумм метаданных (ежедневно)
- Deep scrub: побитовое сравнение реплик (еженедельно)
- DOT-схема: scrub flow — primary сравнивает свои данные с репликами
- Что происходит при расхождении: inconsistent PG → repair
- Настройка: `osd_scrub_begin_hour`, `osd_scrub_end_hour`, `osd_deep_scrub_interval`

### 5.6. Практикум: трассировка записи *(4 стр.)*
- `ceph daemon osd.X perf dump` — смотрим счётчики операций
- `ceph daemon osd.X config set debug_osd 20` — включаем отладку
- `rados put` объект и трассируем его путь по логам OSD
- Сопоставляем с теоретической схемой из §5.3
